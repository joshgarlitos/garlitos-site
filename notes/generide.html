generide: Procedurally Generating Roller Coasters for OpenRCT2

python OpenRCT2 genetic algorithms reverse engineering game development

Last updated: July 2026

generide is a Python tool for generating roller coaster track files that load directly into OpenRCT2. The long-term goal is a genetic algorithm that evolves coasters players actually want to ride. The near-term goal is more modest: understand the binary format well enough to write valid files from scratch.

I grew up playing RollerCoaster Tycoon 2. OpenRCT2 is an open-source reimplementation that runs on modern hardware. What it doesn't have is a way to programmatically generate coasters. generide is an attempt to build that.

The file format problem

Before generating anything, you have to prove you understand the format. A .td6 file is how OpenRCT2 saves and shares individual ride designs. There's no official spec. What exists is a community-maintained document that's mostly accurate and occasionally just wrong.

The format has two layers. The outer layer is a custom run-length encoding scheme RCT2 has used since 1999: one control byte tells you whether the next chunk is literal data or a single repeated byte. The inner layer is a flat binary structure with values at specific offsets. Ride type sits at 0x00. Train count at 0x4c. Lift hill speed packed into the bottom 5 bits of 0xa2. Some fields are bit-packed because in 1999, every byte mattered.

The architecture

The project is structured as a bottom-up stack, each layer testable in isolation: 

compressed .td6 file
       ↕  rle.py       ← custom RLE compress/decompress
decompressed bytes
       ↕  td6.py       ← byte offsets ↔ Ride dataclass
Ride object
       ↕  geometry.py  ← where does each track piece land?
Validated coaster
       ↕  generator    ← genetic algorithm
A coaster that doesn't fall off the map

Building bottom-up matters here. If I generated a coaster and it failed to load, I'd have no way to know whether the bug was in the generator, the format encoder, or the geometry math. Starting from the binary and working up means each layer can be verified before the next one depends on it.

Where it stands

Phase 1 is done. I can read a real .td6 file exported from OpenRCT2, decode it into a Python dataclass, re-encode it, and get back bytes that match the original. Sixteen tests passing, using a real Mine Train ride as the test fixture, not a synthetic one.

The reference implementation I cross-checked against is a Go project called kevinburke/rct. It had a geometry bug I almost ported: cosdeg() calls math.Sin instead of math.Cos. That kind of error doesn't fail loudly. It makes every generated coaster geometrically wrong by 90 degrees of rotation. I caught it by tracing the math manually rather than treating the reference as ground truth.

Phase 2 is the current focus: track segment geometry, meaning given a piece at a known position and direction, compute where the next piece lands.

Phase Goal Status
1 Round-trip a real .td6 file through Python Done
2 Track segment geometry In progress

What comes later

Phase 3 is hand-authoring a coaster as a Python list of segments, encoding it to TD6, and loading it in-game. Phase 4 is the genetic algorithm.

The interesting problem lives in Phase 4. The game outputs excitement, intensity, and nausea ratings, which gives a computable fitness signal. But optimizing against the game's own ratings is a different problem than optimizing for "would a human want to ride this." That gap is the actual thing worth figuring out.
